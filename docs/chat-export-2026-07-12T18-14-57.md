### ADR 01: Delivery Mechanism via Development Dependency
* **Context:** We need to provide maintainers with automated packaging tools without contaminating their consumers' environments with transitive package/DLL bloat.
* **Decision:** The meta-tool will be distributed as a NuGet package configured with `<DevelopmentDependency>true</DevelopmentDependency>` and `PrivateAssets="all"`. 
* **Consequence:** The meta-tool runs strictly during the maintainer's build and pack phases. The consumer inherits only a raw, dynamically generated XML MSBuild `.targets` script, leaving a zero-weight footprint.

### ADR 02: Adherence to Open Standards (`agentskills.io`)
* **Context:** Agent tools are fragmented, but cross-vendor standardization is emerging.
* **Decision:** The bundled folder structure will rigorously follow the three-tier progressive disclosure scheme defined by the [Agent Skills Open Specification](https://agentskills.io/specification).
* **Consequence:** The internal package assets will map strictly to:
  ```text
  agent-assets/skills/[skill-name]/
  ├── SKILL.md                 # Level 1 (Frontmatter) & Level 2 (Procedural Playbook)
  ├── references/              # Level 3 (Deep technical documentation/API schemas)
  ├── assets/                  # Level 3 (Templates, JSON schemas, examples)
  └── scripts/                 # Level 3 (Diagnostics, automated orchestration helper scripts)
ADR 03: Precision Version-Naming & Workspace IsolationContext: Minor package updates introduce API changes. In a .NET Monorepo, multiple sub-projects might reference different versions of the same library, leading to directory write collisions or dead documentation left behind.Decision: 1. Skill directories and internal YAML names will be dynamically derived from the full semantic version of the assembly, sanitizing illegal characters to match strict naming rules (use-[packageid]-v[semver-with-hyphens]).
2. All assets (docs, examples) are deeply nested inside that specific skill's workspace. They are never written loosely to the root of the consumer's repository.
3. The consumer targets file will inject a .gitignore containing * inside the generated skill subfolder root directory.Consequence: Multiple versions of a package's skill can safely coexist inside a single monorepo. Consumers will never see root-level pollution, and dynamically copied skills will never be accidentally committed to their source control history.ADR 04: Pure Body Scaffolding over Brittle Manual ParsingContext: Maintainers should not be forced to maintain complex, repetitive YAML metadata block headers inside their raw documentation sources.Decision: The default input (AgentSkillBodyFile) expects a pure markdown body. The SDK dynamically generates and prepends the compliant YAML frontmatter block during packaging. Power users can explicitly bypass this behavior via an override variable.3.
